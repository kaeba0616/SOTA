# SOTA 콤보 빌드 최적화기 — 설계 문서

**날짜:** 2026-06-01
**프로젝트:** SOTA (Sephiria Optimal Tablet Arranger)
**상태:** 설계 확정, 구현 계획 대기

---

## 1. 목표

게임 *Sephiria*의 인벤토리에서, **사용자가 보유한 석판·아티팩트만으로 목표 콤보 패밀리 빌드를 최대화하는 최적 배치**를 자동 계산한다.

**입력**
- 게임 인벤토리 **스크린샷 1장** (보유 석판·아티팩트, 중복 개수 포함)
- 목표 **콤보 패밀리 1개** (예: `잉걸불(yinggalbul)`)
- 최대 인벤토리 **슬롯 수 N**

**출력**
- 6열 × ⌈N/6⌉ 그리드에 배치한 **최적 배치도 이미지**
- 텍스트 요약: 달성 점수, 콤보 발동 단계, 각 아티팩트 최종 유효레벨, 근사/제외된 특수 아티팩트 목록

---

## 2. 게임 메커니즘 모델

정확도가 핵심이므로 **검증 가능한 평가엔진**으로 격리한다.

- **그리드**: 6열 고정. 슬롯 수 N → 마지막 줄은 부분적으로 채워짐 (위키 시뮬레이터와 동일하게 하단으로 1칸씩 증가).
- **셀 점유**: 아티팩트는 **전부 1×1** (위키 데이터에 size 필드 없음 확인). 석판은 `size`가 있어 다칸 가능 + 회전(rotatable) + 배치 제약(top/bottom/left_right).
- **석판 효과**: `tablets.json`의 `effects[]`. **두 종류**(데이터 검증, pos XOR shape):
  - **pos 기반** `{pos:[dx,dy], type, value}`: 상대 위치 1칸. `level_add`=유효레벨 +value (음수 가능 — `도래` −1). 좌표: dx=열 우+, **dy=행 위+**. 회전 시 `pos` 회전 변환.
  - **shape 기반(영역)** `{shape, type, value}` (pos 없음): 석판이 놓인 줄/영역 전체. shape ∈ `{row, column, diagonal, top, bottom}`. **약 10개 석판**(기적·응집·전이·정의·기반·동시성·반항·차양·경계·광휘)이 사용 → **평가엔진이 반드시 별도 처리**(pos 환원 불가).
  - **`restriction_remove`**(오라클: 보라색 셀, value=boolean), rotatable/size/restriction 모델링.
  - **geometry 출처(정정)**: 위키 `/large` 석판 페이지에서 hover 시 **5×5 효과 그리드 툴팁**이 렌더 → geometry 추출 가능. 누락 4개(courage/honor/hospitality/peace)는 이 오라클로 작성 완료. 또한 `encouragement`는 stale 아니라 `home_town`(고양)의 rename임을 발견·정정.
- **아티팩트 유효레벨**: `clamp(시작레벨 + Σ(자기 셀을 가리키는 석판 level_add), 1 .. maxLevel)`.
  - **maxLevel은 아티팩트마다 다름** (사용자 확인). `effect.content`의 레벨스케일 항목 수 = 그 아티팩트의 단계 수이며 **2~8개로 제각각** (균일한 4 아님 — 데이터 검증 완료). 즉 "전역 레벨 캡 4" 가정은 폐기.
  - 데이터의 `level` 필드는 시작레벨/캡과 무관하게 보임(0~14 분포) → 의미 미확정, 평가엔진에서 직접 사용하지 않고 content 단계 수로 maxLevel 산정.
- **콤보 임계 보너스**: 인벤토리 내 같은 콤보 아티팩트 개수가 임계치 `2/4/6/8/10`개를 넘으면 단계별 발동 (전 콤보 공통, min 2 / max 10). `combos.json`에서 정의 로드.

### 콤보 패밀리 (19종, 위키 확인)
`잉걸불(yinggalbul)`, `빙하(glacier)`, `마법공학(magic_engineering)`, `견고(firmness)`, `정밀(precision)`, `행성(planet)`, `그림자(shadow)`, `저주(curse)`, `태양검(sun_sword)`, `학회(academy)`, `호수(lake)`, `봄노래(spring_song)`, `수호(guardian)`, `흥정(bargaining)`, `동료(colleague)`, `빙기(ice_weapon)`, `원소(element)`, `엑스트리움(extrium)`, `신비(mystery)`

## 3. 목적함수 (확정)

```
Score = Σ(목표 콤보 아티팩트의 유효레벨)
      + 콤보 임계 보너스(목표 콤보 아티팩트 개수 → 도달 단계)
      − 페널티(배치 불가 / 제약 위반)
```

**근거:** 효과 텍스트(`content`)가 이질적(데미지/치명타%/피격 발동/스킬 획득)이라 수치 합산이 주관적·부정확하다. "유효레벨 합"은 단조(레벨↑ = 무조건 강함)·반박 불가·구현 단순한 지표이며, 검증 오라클이 없을 가능성에도 안전하다. 효과값 완전 파싱 합산은 v2로 보류.

## 4. 특수 아티팩트 처리 (확정: 근사 + 명시)

일부 아티팩트(약 4~5개)는 표준 모델을 벗어난다:
- `검은 비늘`: "인벤토리 내 **석판 개수**에 비례" — 전역 의존.
- `캘세더니 열쇠`: "**놓인 가로줄**에 따라 콤보 부여" — 위치 의존 콤보.

**v1 정책:** 이들을 일반 아티팩트처럼 **근사 처리**(특수 규칙 무시, 자기 레벨/고정 콤보로 취급)하되, 결과 리포트에 *"근사 처리됨"*을 **명시**한다. 조용히 틀린 값을 내지 않는다. 정밀 모델링은 v2.

## 5. 아키텍처 (모듈 분해)

```
sota/
  data/          # 정적 게임 데이터
    artifacts.json   (위키 스크래핑)
    combos.json      (콤보별 임계 보너스 — 별도 스크래핑)
    tablets.json     (수작업 유지 — 기존 tablet.json 정리·보완)
  scrape/        # 위키 RSC 스트림(self.__next_f) → JSON 추출기
  recognize/     # 기존 CNN 래핑: 스크린샷 → 보유 아이템(+개수) + 슬롯 수
  model/         # 도메인 객체: Grid, Tablet, Artifact, Placement, Combo
  evaluate/      # 레이아웃 → 점수 (메커니즘 구현, 단위 테스트 대상)
  solve/         # 유전 알고리즘 솔버
  render/        # 배치도 → 그리드 이미지 (Pillow)
  cli.py         # 파이프라인 진입점
```

핵심 원칙: `evaluate`(점수 정의)와 `solve`(탐색)를 분리 → 솔버를 바꿔도 점수 정의·데이터·렌더링은 재사용.

## 6. 데이터 파이프라인

- **아티팩트** (`artifacts.json`): Playwright 헤드리스로 위키 시뮬레이터를 열어 **React fiber 워크**로 추출 (검증 완료, 248개). `__next_f` RSC는 지연 로딩이라 불안정 → 팔레트 버튼의 `__reactFiber$`에서 상위로 올라가 `data`(len 248) 배열을 읽는다. 필드: `id, value/label_eng, label_kor, tier(common/advanced/rare/legend/solid), effect.sets(콤보키), effect.content(레벨스케일), level(의미 미확정), image, description`. `content`의 슬래시 스케일을 정규식으로 구조화(단계 수 = maxLevel).
- **석판** (`tablets.json`): **효과 geometry(`pos`)는 위키 어디에도 정형 데이터로 없음** (시뮬레이터 팔레트는 `{id,label,image,rotation}`만, 계산 모듈은 별도) → 로컬 `tablet.json`이 수작업 권위 데이터. **게임 석판 = 54개**(시뮬레이터 검증) vs 로컬 50개 → **누락 5개 추가**(courage/home_town/honor/hospitality/peace) + **이름 불일치 6개 매핑**(advent↔arrival, compete↔competition, linear↔goodness, triceps↔three_heads, unity↔fusion, yearning↔desire) + **구버전 추정 1개 확인**(encouragement).
- **콤보 정의** (`combos.json`): **출처 확정** — `/combo` 페이지 RSC. 스키마: `{key, label, image, minCount, maxCount, effects:[{count, effect}]}`. **콤보 19종** 전부, 아티팩트 `effect.sets` 키와 정확히 일치. 임계단계 = `2/4/6/8/10`개.
- **데이터셋 범위 확정**: canonical 아티팩트 셋 = **위키 전체 248개(상위집합)**. (시뮬레이터 데이터 배열 기준 248, 전부 활성; 사용자가 눈으로 본 "21×8≈168"은 팔레트 시각 추정치였음.) 보유 여부는 CNN 인식이 결정하므로 DB가 상위집합이어도 무해.
- **id 정규화 레이어 (신규)**: 위키 키 ↔ 로컬 이미지 파일명 ↔ tablet.json 간 명명 불일치가 아티팩트·석판·이미지 전반에 존재 → **canonical id 매핑 테이블**을 둬서 세 출처를 연결.
- **에셋 동기화**: 로컬 아티팩트 이미지 237 vs 위키 248. 이름 정규화로 다수 매칭(lightningboomerald↔lightning_boomerald 등), 진짜 누락 약 11개(academy_fountain_pen, defect_probe, heart_of_the_beast, plasma_helmet, plump_fruits, red_planet_observation_log, snow_mountain_big_eared_bat, stone_wheel, thunder_judgment, unalloyed_gold_needle, zorous_watering_can)를 `img.sephiria.wiki`에서 받아 CNN 재학습.
- **버전 고정**: 위키 버전(현재 v0.12.0) 기록 → 패치 시 재추출.

## 7. 인식 파이프라인

- 기존 `CNN/inven_test.py`를 `recognize/`로 정리: 스크린샷 → (보유 석판 목록+개수, 보유 아티팩트 목록+개수).
- 슬롯 수 N: 인식 또는 사용자 인자.
- **인식 결과 확인/수정 단계** 권장 (CNN 오류가 솔버 입력을 오염시키므로).
- 결과를 `model` 객체로 변환 → 솔버 입력.

## 8. 솔버 — 유전 알고리즘 (확정)

이 문제는 2차 배치(석판 효과가 상대 위치 의존) + 부분집합 선택이 결합된 NP-hard.

- **염색체**: 길이 = 배치 후보 아이템 수. 각 유전자 = `(셀 인덱스 | 미배치, 회전상태)`. **보유 > 슬롯**일 수 있으므로 "미배치" 상태로 **부분집합 선택**을 표현. 중복 아이템은 개수만큼 별도 유전자.
- **적합도**: §3 평가엔진 점수.
- **연산자**:
  - 선택: 토너먼트
  - 교차: 위치 보존 교차(PMX 계열) — 좋은 부분 배치(석판+아티팩트 클러스터)를 충돌 없이 교환
  - 변이: 아이템 이동 / 두 아이템 스왑 / 석판 회전 / 배치↔미배치 토글
  - 수선(repair): 셀 중복·제약 위반(석판 size/restriction)을 변이 후 정리
- **엘리트 보존** + 다세대 반복, 정체 시 조기 종료.
- 라이브러리: `DEAP` 또는 경량 직접 구현.
- (선택) 검증용: 소규모 인스턴스에서 CP-SAT 정확해와 대조해 GA 품질 확인.

## 9. 출력 / 렌더링

- `render/`: 최적 배치를 위키 시뮬레이터 유사 **그리드 이미지**(Pillow)로. 각 셀에 아이콘, 석판 효과 방향, 아티팩트 최종 유효레벨 표시.
- 텍스트 요약: 달성 점수 / 콤보 발동 단계 / 아티팩트별 최종 레벨 / **근사·제외된 특수 아티팩트 목록**.

## 10. 단계별 구현 로드맵

1. **데이터 확보** — fiber 스크래퍼로 `artifacts.json`(248) + `combos.json`(19) 생성, `tablets.json` 정리(누락 5 추가·이름 6 매핑·구버전 1 확인), **id 정규화 매핑** 작성, 누락 이미지 11개 동기화·CNN 재학습.
2. **도메인 모델 + 평가엔진** — 레이아웃→점수, 단위 테스트(메커니즘 검증).
3. **GA 솔버** — 보유 아이템·콤보 입력 → 최적 배치 (선택·배치·회전·수선).
4. **인식 연결** — CNN 스크린샷 → 보유 아이템(+개수) → 솔버, 확인 단계 포함.
5. **렌더링 + CLI** — 한 명령으로 스크린샷+콤보 → 배치 이미지 + 요약.
6. **(선택) v2** — 효과값 완전 파싱, 특수 아티팩트 정밀 모델링, CP-SAT 정확해 모드.

## 12. 검증 전략 (오라클)

위키 시뮬레이터가 배치에 대한 최종 아티팩트 레벨/효과를 **계산해 표시**한다(사용자 확인). 이를 평가엔진의 정답지로 사용한다:

1. 대표 배치 케이스(단일 석판+아티팩트, 음수 석판, 다중 중첩, 콤보 임계 경계)를 시뮬레이터에 입력.
2. 시뮬레이터의 표시값을 추출(React fiber / DOM)하여 **골든 테스트** 데이터셋 구성.
3. 평가엔진 출력이 골든값과 일치하는지 단위 테스트로 강제 → 메커니즘 미세규칙(레벨 캡, restriction, 음수)을 경험적으로 확정.
